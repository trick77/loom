Loom Prompt Classifier

Dynamic system-prompt enrichment driven by a per-conversation category, classified for free on the first message.

1 · Context & goal

Loom sends one static base system prompt to every conversation. The goal is to make the assistant think differently and surface different unasked-for value depending on what kind of request the conversation is.

Where Perplexity's classifier answers "how should I format this?", Loom's answers "how should I think about this, and what should I surface that the user didn't ask for?" — the enrichment gap. Perplexity would handle "who was Einstein" with a generic People template (short bio, done); Loom answers the bio and proactively points toward gravitational-wave detection or the latest relativity-adjacent research.

The win: Loom already makes a cheap, thinking-disabled utility LLM call on the first message to generate the thread title. We add a second, separate utility call — also off the first message, also thinking-disabled — that returns just the category, run concurrently with the title call (added latency ≈ one call). Keeping it separate from titling leaves the carefully-tuned title prompt untouched (no title-quality regression) and lets the classify prompt be purpose-built for accuracy; cost is negligible (tiny input + ~3–5 output tokens). The category is stored on the thread and drives a small, lean instruction block injected into the system prompt for every turn, and shows up as a pill in the status line.

Note on caching: the title and classify calls use different system prompts, so the shared first-message text won't yield a prompt-cache hit — fine, the classify call is cheap in absolute terms regardless.

Decisions locked with the user

  • Classify once, piggybacked on the existing title call → thread-level, fixed for the conversation (re-classification on drift deferred to v2).
  • Single category + general fallback — the model is never forced to mis-bucket.
  • Named "prompt classifier"; DB/API field is category (not "semantic router" — that term implies embedding-based routing, which this isn't).
  • Fully automatic — no manual category picker (matches the no-button UX preference).

2 · Taxonomy

Anchored on proven leak data. The base is Perplexity's leaked "Query type specifications" (10 types). Perplexity uses them to decide how to format; Loom reuses the same proven partition of what-people-ask but swaps each rule from format → think + enrich. Perplexity is search-first, so it omits the categories that dominate general assistant usage — Loom adds those.

Every block has two parts: (a) FORMAT — how to shape the answer; (b) THINK/SURFACE — how to think about it and what to surface unasked. Utility categories carry FORMAT only. Register: lean and filler-free, but complete directives (the knowledge_discovery block sets the bar).

Sourcing: the 10 Perplexity types take their FORMAT verbatim from the leak; the THINK/SURFACE half is always Loom's design (that's the enrichment gap). The 7 Loom-only categories are marked [Loom].

3 · Static vs. dynamic prompt — conflict resolution

The current static base prompt hard-mandates prose and forbids lists ("Do NOT use bullet points or numbered lists… When in doubt, use prose."). This clashes with list-shaped categories (cooking_recipes, how_to, coding, shopping).

Resolution (per user — pull clash-risky formatting out of static, push it into the dynamic blocks where lists make sense):

  • Soften the static base: keep a light prose default for uncategorized/general chats, but remove the hard "no bullet/numbered lists" prohibition so it can't override a category block.
  • Each category block owns its formatting — the injected block is authoritative for that conversation's shape.
  • Relocate: the static base's brainstorming rule ("at most 12 options and recommend one") moves into the brainstorming block. Tool-routing rules (file-creation, image-generation) stay in static.

4 · Implementation

Branch from fresh origin/master. Local master was 16 commits behind; the metrics restyle (73d9aa5) and Loom rebrand live there. Work happens in a git worktree off freshly fetched origin/master.

4.1 Classifier call — a separate utility call, concurrent with the title call

  • New shared package backend/internal/classifier: the 17-value enum, the literal injected block per category, a PromptGuide() for the classifying prompt, plus Valid / Normalize / Block. Single source of truth, imported by both llm and httpapi.
  • Title call untouched: GenerateThreadTitle (llm/thread_title.go) keeps its proven prompt and (string, error) return — no regression risk.
  • New ClassifyThread(ctx, userMessage) (llm/thread_classify.go): system prompt = a short classify instruction + classifier.PromptGuide(); the model replies with only one category value (not JSON, ~3–5 tokens). Reuses the existing executeUtilityChatRequest path (thinking: disabled, 32-token cap is ample). Parse = trim + classifier.Normalize → invalid/empty → general; request error → general (never block the answer).
  • Concurrency: generateAndSendThreadTitle runs the title and classify calls concurrently, awaits both, then writes title + category in one UpdateThread. Added latency ≈ max(title, classify).

4.2 Persist category on the thread

  • Migration 0013_thread_category.sql: ALTER TABLE threads ADD COLUMN category TEXT NOT NULL DEFAULT ''.
  • chat.Thread.Category, UpdateThreadInput.Category, scanThread, both thread SELECT lists, and the UPDATE SET clause all carry category.
  • generateAndSendThreadTitle persists title + category in one update, emits the existing thread SSE event (now carrying category), and returns the category so the first turn can inject the block immediately (the call is synchronous, before the answer history is built).

4.3 Inject the block into the system prompt

  • buildLLMHistory gains a classifierContext block appended exactly like the existing user/project/knowledge/document blocks. The handler passes classifier.Block(category) — enrichment categories get both parts, utility categories FORMAT-only, general/unknown nothing.

4.4 Status-line pill

The metrics line was just restyled on master (73d9aa5). buildMetricsString stays text-only; the category is a pill rendered as its own badge in MessageMetrics.tsx, to the left of the metrics span, sourced from thread state (single source of truth — correct even on the first message once the thread SSE event lands). category?: string is added to the Thread type. A humanizer turns knowledge_discovery → "Knowledge discovery".

6 · Risks & open notes

  • Classification accuracy (the main remaining risk): a 17-way pick in one fast, thinking-disabled call. A purpose-built classify prompt (separate from titling) helps; the general fallback guards parse failure, not confident mis-classification. If routing is noisy: shrink the taxonomy, or give this one call a small reasoning budget. (Title-quality regression is no longer a risk — the title call is untouched.)
  • Deferred to v2: re-classification on topic drift; a sensitive caveat class.
